# Issue Authoring Guide

Use this repo's issue templates together with the draft backlog in `issue-drafts/`.

## What works well for Wave-style issues

- one clear problem per issue
- small or medium scope
- explicit acceptance criteria
- exact files likely to change
- a concrete verification step
- labels that help contributors self-select

## Recommended labels

- `good first issue`
- `help wanted`
- `backend`
- `frontend`
- `docs`
- `tests`
- `design`
- `enhancement`

## Suggested complexity split

- `Trivial`: docs, copy, small UI polish, validation tweaks
- `Medium`: isolated feature work in one layer
- `High`: cross-cutting changes, API plus UI, larger refactors

## Before publishing an issue

- confirm the task still matters
- confirm the code references are still correct
- decide whether tests are required
- decide whether screenshots or short videos would help review
